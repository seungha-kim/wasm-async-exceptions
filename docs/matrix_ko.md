## Resolve-only C++ exception stress 시나리오

S5-S17은 첫 매트릭스 이후 JS Promise rejection을 실험에서 제거하기 위해 추가했다. 모든 controlled Promise는 resolve된다. 따라서 failure는 JS rejection이 Wasm boundary를 건너는 문제가 아니라 C++ exception/suspend 상호작용을 가리킨다.

이 시나리오들은 A/B/D에서만 실행한다.

- A: 실용적인 Asyncify + JS exception emulation baseline.
- B: Asyncify + Wasm EH. EH 축만 바꾼 영향을 격리하는 데 유용하지만, 현재는 fragile intermediate path로 문서화한다.
- D: JSPI + Wasm EH. 런타임 표준 기능을 모두 쓰는 경로.

## 관례

- Playwright spec이 성공적으로 실행된 뒤에만 셀을 채운다.
- pitfall reproduction, 즉 "expected" failure에서는 정확한 console sequence와 그 mechanism을 설명하는 `EXPECT.md` 링크를 함께 기록한다.
- `WARN`은 scenario가 명시적인 `PASS:*:done` 라인에 도달했지만, clean success로 보아서는 안 되는 post-done `[pageerror]`를 냈다는 뜻이다.
- 여기서 참조하는 `PASS` / `FAIL` 라인은 Wasm 쪽 `console.log("PASS:<id>")` / `console.log("FAIL:<id>")` 신호다.
- `[pageerror]` 라인은 Playwright가 포착한 JS `uncaughtException` event이며, catch boundary 밖으로 새어 나온 Wasm throw를 표면화한다.
